# Story 7.3: Extension App Scaffold

Status: done

## Story

As a developer,
I want a properly configured browser extension workspace in the pnpm monorepo,
So that I can develop the extension with modern tooling consistent with the rest of the project.

## Acceptance Criteria

1. **AC1 (WXT workspace):** `apps/extension/` is a valid pnpm workspace package (`@battlecrm/extension`). WXT is installed and configured with React + TypeScript + Tailwind v4 CSS in `apps/extension/`. The manifest (generated by WXT) targets Manifest V3. `pnpm --filter @battlecrm/extension build` succeeds from the repo root.

2. **AC2 (Manifest permissions):** The WXT-generated manifest includes:
   - `permissions`: `["storage", "activeTab", "scripting"]`
   - `host_permissions`: `["*://www.linkedin.com/*"]`
   - Content script injected on `*://www.linkedin.com/in/*`

3. **AC3 (Stable extension ID):** `wxt.config.ts` includes a `manifest.key` field (base64 RSA public key) so the extension keeps a stable Chrome ID across dev reloads. This allows `EXTENSION_ORIGINS` in `.env` to remain stable.

4. **AC4 (Biome):** `pnpm biome check .` from repo root passes on `apps/extension/**` files (Biome root config already covers `apps/**`). No separate ESLint/Prettier is used.

5. **AC5 (Entrypoints structure):** The following WXT entrypoints exist as working stubs (build succeeds, no runtime logic required for this story):
   - `src/entrypoints/background.ts` — service worker stub
   - `src/entrypoints/content.ts` — content script stub matching `*://www.linkedin.com/in/*`
   - `src/entrypoints/popup/index.html` + `App.tsx` — action popup stub (renders "BattleCRM" heading)
   - `src/entrypoints/panel/index.html` + `App.tsx` — floating window stub (renders "Panel" heading)

6. **AC6 (Lib stubs):** The following utility skeletons exist in `src/lib/`:
   - `api.ts` — typed fetch wrapper with Bearer auth signature (stub body only)
   - `storage.ts` — `chrome.storage.local` wrapper with typed interface (stub body only)
   - `linkedin.ts` — DOM scraping function signatures (stub body only)

7. **AC7 (Root scripts):** `package.json` at repo root has:
   - `"dev:extension": "pnpm --filter @battlecrm/extension dev"` — starts WXT dev server with HMR
   - `"build:extension": "pnpm --filter @battlecrm/extension build"` — produces production build

8. **AC8 (Build output):** After `pnpm build:extension`, `apps/extension/.output/chrome-mv3/` exists and contains a loadable unpacked extension (manifest.json + background service worker + content scripts + popup/panel HTML).

9. **AC9 (Shared types):** `apps/extension/` imports `@battlecrm/shared` as a workspace dependency. No TypeScript errors from shared type imports. `pnpm --filter @battlecrm/extension type-check` passes.

10. **AC10 (Validation):** `pnpm biome check --write .` → 0 errors. `pnpm --filter @battlecrm/extension build` → success. `pnpm build:extension` from root → success.

### Task 6: Lib utility skeletons (AC6)

- [x] **6.1** Create `apps/extension/src/lib/storage.ts`:
  ```typescript
  export interface ExtensionStorage {
    token?: string
    baseUrl?: string
    email?: string
  }

  /** Read all auth-related keys from chrome.storage.local */
  export async function getStorage(): Promise<ExtensionStorage> {
    return browser.storage.local.get(['token', 'baseUrl', 'email']) as Promise<ExtensionStorage>
  }

  /** Persist partial auth data to chrome.storage.local */
  export async function setStorage(data: Partial<ExtensionStorage>): Promise<void> {
    return browser.storage.local.set(data)
  }

  /** Delete token, baseUrl, email (called on logout or 401) */
  export async function clearAuth(): Promise<void> {
    return browser.storage.local.remove(['token', 'baseUrl', 'email'])
  }
  ```

  ```typescript
  import type { ExtensionCheckResponse, ExtensionProspectData } from '@battlecrm/shared'

  /** POST /api/extension/auth/login — returns raw token (one-time display) */
  export async function loginExtension(
    baseUrl: string,
    email: string,
    password: string,
  ): Promise<{ token: string; user: { id: string; email: string } }> {
    const res = await fetch(`${baseUrl}/api/extension/auth/login`, {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({ email, password, name: 'Mon Chrome' }),
    })
    if (!res.ok) throw new Error(`HTTP ${res.status}`)
    return res.json()
  }

  /** POST /api/extension/auth/logout — revokes the token */
  export async function logoutExtension(baseUrl: string, token: string): Promise<void> {
    await fetch(`${baseUrl}/api/extension/auth/logout`, {
      method: 'POST',
      headers: { Authorization: `Bearer ${token}` },
    })
  }

  /** GET /api/extension/prospects/check */
  export async function checkProspect(
    baseUrl: string,
    token: string,
    linkedinUrl: string,
  ): Promise<ExtensionCheckResponse> {
    const res = await fetch(
      `${baseUrl}/api/extension/prospects/check?linkedin_url=${encodeURIComponent(linkedinUrl)}`,
      { headers: { Authorization: `Bearer ${token}` } },
    )
    if (!res.ok) throw new Error(`HTTP ${res.status}`)
    return res.json() as Promise<ExtensionCheckResponse>
  }

  /** POST /api/extension/prospects */
  export async function createProspect(
    baseUrl: string,
    token: string,
    payload: { name: string; linkedin_url: string; company?: string; title?: string; email?: string; phone?: string },
  ): Promise<ExtensionProspectData> {
    const res = await fetch(`${baseUrl}/api/extension/prospects`, {
      method: 'POST',
      headers: { Authorization: `Bearer ${token}`, 'Content-Type': 'application/json' },
      body: JSON.stringify(payload),
    })
    if (!res.ok) throw new Error(`HTTP ${res.status}`)
    return res.json() as Promise<ExtensionProspectData>
  }

  /** PATCH /api/extension/prospects/:id */
  export async function updateProspect(
    baseUrl: string,
    token: string,
    id: string,
    payload: Partial<Omit<ExtensionProspectData, 'id' | 'linkedinUrl' | 'funnelStageId' | 'funnelStageName'>>,
  ): Promise<ExtensionProspectData> {
    const res = await fetch(`${baseUrl}/api/extension/prospects/${id}`, {
      method: 'PATCH',
      headers: { Authorization: `Bearer ${token}`, 'Content-Type': 'application/json' },
      body: JSON.stringify(payload),
    })
    if (!res.ok) throw new Error(`HTTP ${res.status}`)
    return res.json() as Promise<ExtensionProspectData>
  }

      headline: '',
      company: '',
      canonicalUrl: normalizeLinkedInUrl(location.href),
    }
  }

  /**
   * Normalize a LinkedIn profile URL — strips query params, hash, and trailing slash.
   * Used in both content.ts (before sending CHECK_PROSPECT) and background.ts (before API call).
   */
  export function normalizeLinkedInUrl(url: string): string {
    try {
      const parsed = new URL(url)
      parsed.search = ''
      parsed.hash = ''
      return parsed.toString().replace(/\/$/, '')
    } catch {
      return url.replace(/[?#].*$/, '').trim().replace(/\/$/, '')
    }
  }
  ```

## Dev Notes

### Why WXT (not Plasmo, not raw webpack/vite)

Architecture decision validated 2026-03-01: WXT is the current leader in browser extension tooling with:
- **Vite-based** (same tooling philosophy as `apps/frontend`)
- **Cross-browser** (Chrome MV3 + Firefox via single config)
- **~43% smaller bundle** than Plasmo
- **React module** (`@wxt-dev/module-react`) for seamless React integration
- **Auto-imports** for WXT globals (`defineContentScript`, `defineBackground`, `browser.*`)

[Source: architecture.md → "Browser Extension Architecture (Epic 7)" → "Technology Stack"]

### Stable Extension ID via `key` field (AC3)

**Why this matters:** Chrome assigns a random extension ID each time you load an unpacked extension. If the ID changes, `EXTENSION_ORIGINS` in `.env` (used by the backend CORS config) must be updated on every reload — impractical for dev.

**How to generate the key once:**

```bash
# From apps/extension/ directory
openssl genrsa 2048 | openssl pkcs8 -topk8 -nocrypt -out key.pem
openssl rsa -in key.pem -pubout -outform DER | openssl base64 -A
```

Copy the base64 output into `wxt.config.ts` as `manifest.key`. Add `key.pem` to `.gitignore` — do NOT commit the private key.

The resulting stable extension ID will be shown in `chrome://extensions` after loading unpacked. Add it to `.env`:
```
EXTENSION_ORIGINS=chrome-extension://<YOUR_STABLE_ID>
```

[Source: architecture.md → "CORS Configuration"]

### WXT auto-imports — no manual chrome.* imports needed

WXT provides global auto-imports for:
- `browser.*` — cross-browser compatible polyfill (use this, NOT `chrome.*` directly)
- `defineBackground()` — for `background.ts`
- `defineContentScript()` — for `content.ts`
- WXT handles injecting the proper chrome/browser type declarations via `.wxt/tsconfig.json`

**Never import from `webextension-polyfill` manually** — WXT provides this automatically.

### Tailwind v4 in WXT — no config file needed

Tailwind v4 works via the `@tailwindcss/vite` plugin (same as `apps/frontend`). No `tailwind.config.js` needed. Just:
1. Add `tailwindcss()` to `vite.plugins` in `wxt.config.ts`
2. Create `src/assets/tailwind.css` with `@import "tailwindcss";`
3. Import this CSS in each React App.tsx: `import '../../assets/tailwind.css'`

Do NOT use `@tailwind base;` / `@tailwind utilities;` — those are Tailwind v3 directives.

[Source: apps/frontend/package.json — uses `@tailwindcss/vite ^4.1.18`]

### `browser.*` vs `chrome.*` — use browser everywhere

WXT re-exports `browser` as a global that works on both Chrome and Firefox. Using `chrome.*` directly would break Firefox builds. The architecture requires both Chrome and Firefox support (NFR68).

Exception: `chrome.windows.create` has no direct `browser.windows.create` equivalent in WXT types in some versions — check the WXT docs if you hit a typing issue. The functionality exists in both browsers; it's a typing gap.

### `@battlecrm/shared` in the extension

The extension imports types from `@battlecrm/shared` just like the frontend:
- `ExtensionProspectData`, `ExtensionCheckResponse` — for typed API responses in `lib/api.ts`
- These are already exported from `packages/shared/src/types/extension.ts` (added in Story 7.2)

Since WXT uses Vite under the hood, workspace package resolution works identically to the frontend. No additional configuration needed — `@battlecrm/shared` in `package.json` dependencies is sufficient.

[Source: apps/backend/tests/functional/extension/prospects.spec.ts — pattern for using shared types]

### Message passing architecture — already defined (implement in Stories 7.5/7.6)

The full message passing protocol between content script → service worker → popup/panel is designed and documented in the architecture. This story only creates STUBS — do NOT implement message handlers yet. The stub entrypoints just need to build successfully.

```
Content Script → chrome.runtime.sendMessage({ type: 'CHECK_PROSPECT', linkedinUrl })
Service Worker → reads storage → calls API → updates badge → caches result
Panel → chrome.runtime.sendMessage({ type: 'GET_PANEL_DATA' }) → returns cached result
```

[Source: architecture.md → "Flux de Données"]
