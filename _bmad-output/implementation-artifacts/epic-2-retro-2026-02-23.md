## 🏆 Ce qui a bien marché

### 1. Code Review adversariale — encore indispensable
Systématique sur les 4 stories, elle a attrapé des bugs non détectables par les tests automatiques :
- Story 2.3 : API errors via `toast.error()` au lieu d'inline (**H1**), `<main>` imbriquées, position badges stale pendant DnD
- Story 2.4 : `{{count}}` clé réservée i18next → pluralisation involontaire, test max-15 cassé (assumption 10 stages → réalité 9)

### 2. Pattern two-step reorder (Story 2.2)
Solution élégante pour contourner la contrainte unique PostgreSQL `(user_id, position) WHERE deleted_at IS NULL` : déplacer toutes les positions vers ≥10000 en étape 1, puis assigner les positions finales en étape 2. Réutilisable dans tous les futurs epics avec ordering.

### 4. Croissance des tests sans flakiness

### 5. Qualité et design général

## ⚠️ Ce qui a posé problème

### 1. Bugs d'assumption sur les données de test (pattern récurrent)
- **Story 2.1** : données résiduelles en DB faussaient les tests → ajout de `group.setup`
- **Story 2.4** : test assumait 10 stages par défaut → réalité 9 → test cassé

**Root cause** : les assumptions sur les données sont prises dans la documentation plutôt que vérifiées directement dans le code source (ex: `DEFAULT_FUNNEL_STAGES.length`).

**Accord d'équipe** : toujours vérifier les constantes directement dans le code source lors de l'écriture des tests.

### 2. Pièges des librairies tierces
- **Biome v2** : `noStaticOnlyClass` (Story 2.1), chaining multi-lignes obligatoire (Story 2.4)
- **i18next** : `{{count}}` est une clé réservée qui déclenche la pluralisation (Story 2.4) → renommé `{{total}}`
- **adonis-lucid-soft-deletes** : `withTrashed()` non déclaré dans les types TypeScript → module augmentation `soft_deletes.d.ts` (Story 2.2)

Ces pièges sont documentés au fil de l'eau dans les story files mais pas centralisés.

### 3. Inconsistances de documentation légères
- FR39 dans `epics.md` dit "10 default funnel stages" → réalité : 9 (changé en cours de Story 2.1)
- Stories 3.1, 4.1, 5.1, 7.1 dans `epics.md` disent encore "Row Level Security" → doit être `forUser()` scope

Ces inconsistances sont acceptables à ce niveau d'amplitude, mais doivent être corrigées avant de démarrer Epic 3.

## 💡 Insights clés

1. **La code review adversariale est non-négociable** — elle attrape ce que les tests automatiques ne voient pas, notamment les bugs UX/frontend
2. **Les assumptions sur les données de test doivent toujours être vérifiées dans le code source**, pas dans la doc ou les commentaires
3. **Les changements de spec légers en cours de dev = bonne agilité**, pas un problème d'organisation — tant que ça reste à cette amplitude
4. **La documentation se dégrade discrètement** — un passage de correction avant chaque epic est nécessaire et rapide
