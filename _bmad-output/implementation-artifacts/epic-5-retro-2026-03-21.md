## Ce qui a bien marché

### 1. `hardResetTestData` — fin de l'accumulation E2E

En Epic 4, l'accumulation de données archivées entre les runs de tests était notée comme "acceptable pour le MVP". En Story 5.x, un endpoint de reset hard-delete a été introduit. Les tests repartent maintenant réellement de zéro à chaque run — plus de faux positifs dus à des données résiduelles.

**Pattern établi :** `hardResetTestData(context.request)` en `beforeAll` dans chaque spec. À maintenir pour tous les epics suivants.

### 2. Parallélisation E2E — réduction de 3-4x du temps de run

Avant Epic 5 : `workers: 1`, run séquentiel, ~3 minutes. Après : 4 workers en parallèle, chaque spec fichier tourne dans son propre contexte utilisateur (`e2e-worker-{n}@battlecrm.test`), auth par worker-fixture. Les 11 specs s'exécutent en parallèle sans conflit de state DB.

**Impact direct :** feedback loop E2E 3-4x plus rapide. Investissement ponctuel, bénéfice permanent.

### 3. ProspectTimeline unifiée — la chaîne est complète

La décision de timeline unifiée (Epic 4 retro) a été implémentée avec élégance : interactions et stage transitions mixées, triées par date DESC, avec distinction visuelle par type. L'utilisateur voit l'historique complet d'un prospect en un coup d'oeil — c'est la vision produit originale de FR25.

**Bonus :** edit-in-place sur chaque interaction dans la timeline (non prévu, mais logique) — évite d'ouvrir une page dédiée pour corriger une note.

### 4. Migration 0007 — correction du bon modèle de données

`positioning_id` sur la table `prospects` était un vestige d'une conception prématurée : un prospect n'a pas *un* positioning fixe, il a un positioning par interaction. La migration mid-epic pour supprimer cette colonne était courageuse (breaking change, cascade de mises à jour) mais correcte. Le modèle de données reflète maintenant la réalité métier.

### 5. Mobile navbar — UX proactive

Réfonte du navbar pour mobile non planifiée, réalisée en Story 5.5. L'application est utilisable sur téléphone — cas d'usage clé : logger une interaction juste après un rendez-vous. Infrastructure UX préparée avant d'en avoir besoin.

### 6. 6/6 action items Epic 4 suivis

Premier epic avec suivi parfait des engagements. Les patterns (serializer, shared types, SoftDeletes checklist, UUID validation) sont devenus des réflexes qui n'apparaissent plus en code review.

## Ce qui a posé problème

### 1. Scope expansion non tracé — stories 5.4 et 5.5

Stories 5.4 et 5.5 contiennent des changements de l'ampleur d'une ou deux stories complètes :
- **Story 5.4 :** refonte Accordion → Table pour ProspectsList et PositioningsList, KanbanCard button, AppNavbar logo, mise à jour de 6 fichiers E2E
- **Story 5.5 :** mobile navbar, settings page refactor, restore endpoint, migration 0007, edit-in-place timeline, `useInteractionEdit` hook

Ces changements sont bons et cohérents, mais ils rendent la code review difficile (diff multi-sujets) et l'estimation rétrospective impossible.

**Pattern émergent :** Romain corrige et améliore "au fil de l'eau" plutôt que de différer. C'est une stratégie valide (évite la dette accumulée), mais génère du bruit dans les stories officielles.

### 2. Migration mid-epic = risque de coordination

`migration 0007` (suppression de `positioning_id` sur `prospects`) est une breaking change sur le schéma en plein milieu d'un epic actif. Dans une équipe multi-développeurs, cela bloquerait toutes les branches en cours. Ici c'est gérable car travail principalement solo ou séquentiel — mais le risque augmente avec l'équipe.

### 3. Deux agents en parallèle — patterns potentiellement désynchronisés

La stratégie "un agent sur un gros refacto, un autre sur un coin indépendant" est efficace en temps — mais si les deux modifient des conventions partagées (composants UI, patterns de test), le résultat peut diverger. À surveiller à mesure que la codebase grandit.

### 4. API calls intempestifs — cache TanStack Query sous-optimal

Constaté à l'usage : certains appels API se refont alors que les données n'ont pas changé. Cause probable : staleTime à 0 par défaut, absence de cache key granulaire sur certains endpoints (ex: funnel stages rechargés à chaque ouverture de dialog). Non bloquant, mais impactant la fluidité perçue — à adresser entre Epic 5 et Epic 7.

## Insights clés

1. **"Au fil de l'eau" vs "between epics" — les deux ont leur place.** Les petits correctifs en cours d'epic gardent le code propre en continu. Mais les refactos transversaux (refonte Table, mobile navbar) méritent d'être explicitement sortis des stories pour garder les diffs lisibles.

2. **La correction du bon modèle vaut mieux que le patch de surface.** A5 (afficher le nom du positioning dans ProspectDetail) semblait être un petit fix — ça a déclenché la réalisation que le modèle était faux. Supprimer `positioning_id` des prospects est la bonne réponse, pas l'ajouter à la vue.

3. **E2E parallelization — investissement à faire tôt.** Plus on attend, plus il y a de specs à adapter. Le faire maintenant, avec 11 specs et un pattern clair, était le bon moment.

4. **Epic 7 avant Epic 6 — décision technique et produit alignées.** L'extension LinkedIn est indépendante d'Epic 6, dépend uniquement d'Epic 1 (auth) + Epic 3 (prospects), et représente la plus haute valeur différenciante du produit. Aucune raison architecturale de l'attendre.

## Plan d'investigation — API calls intempestifs

Romain a défini l'ordre d'investigation pour le cache TanStack Query :

## Preview Epic 7 — LinkedIn Browser Extension

### Ce qu'on sait

- **6 stories** : token auth backend, API prospect dédiée extension, scaffold extension, settings/auth UI, badge LinkedIn, floating panel add/update
- **Stack prévue** : extension Chromium (Manifest V3), framework léger (Preact ou vanilla), communication avec l'API BattleCRM via token Bearer
- **Dépendances** : Epic 1 auth backend ✅, Epic 3 prospects API ✅
- **Indépendant de** : Epic 6 (analytics), Epic 8 (CSV import)

### Risques à anticiper

- **Manifest V3 contraintes** : pas de background persistent, service workers limités — à valider tôt en Story 7.1
- **CORS + cookie vs token** : l'extension ne peut pas utiliser les httpOnly session cookies du frontend. Il faudra un mécanisme de token Bearer dédié (Story 7.1). Architecture distincte de l'auth web
- **Tests E2E extension** : Playwright supporte `--load-extension` sur Chromium — configuration spécifique à scaffolder
- **Détection LinkedIn** : scraping du DOM LinkedIn est fragile (classes obfusquées, structure change). Stratégie à définir en Story 7.5

### Préparation requise avant Story 7.1

- Lire la spec Epic 7 dans `epics.md` (stories 7.1 à 7.6)
- Valider le mécanisme d'auth token (durée de vie, révocation, stockage dans `chrome.storage.sync`)
