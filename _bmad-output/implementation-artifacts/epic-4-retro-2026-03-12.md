## Ce qui a bien marché

### 1. Cohérence UI/code — direction d'équipe explicite

Le pattern Accordion shadcn, introduit pour ProspectsList en Epic 3, a été réutilisé directement pour la liste des positionings (Story 4.3) et a déclenché un refacto bénéfique de `ProspectRow/ProspectsList`. Résultat : deux pages de l'application partagent la même base de composants — l'utilisateur navigue sans rupture visuelle.

**Principe validé :** cohérence UI/code n'est pas un accident, c'est une direction. Elle doit être documentée comme contrainte d'architecture pour les prochains epics.

### 2. Pattern bout-en-bout complet

Créer → éditer → archiver → restaurer → lier à un prospect. Aucune demi-feature. L'Epic 4 livre une entité entièrement utilisable dès maintenant, y compris le restore (endpoint émergent non prévu en Story 4.5, ajouté en cours d'implémentation).

### 3. UUID_REGEX extrait en helper réutilisable

`UUID_REGEX` était inline dans le controller Story 4.2 — extrait vers `#helpers/regex` en cours de story. Maintenant réutilisable par tous les controllers futurs. Petit geste, grande valeur.

### 4. Suivi des engagements Epic 3 — 5/6

Troisième epic de suite avec un excellent suivi des action items. Les patterns établis (serializer, shared types, UUID validation) sont devenus des réflexes — ils n'apparaissent plus comme des surprises en code review.

### 5. E2E Playwright : 3 suites stables

Chaque suite isolée avec son propre `beforeAll/resetPositionings`. Pattern serial + reset. Filet de sécurité solide qui n'a laissé passer aucun bug en production.

## Ce qui a posé problème

### 1. Bug SoftDeletes + sous-ressources — récidive

En Story 4.2, l'endpoint `GET /api/positionings/:id/prospects` manquait `.withTrashed()` pour les positionings archivés. Exactement le même bug qu'en Epic 3 (Story 3.6). Corrigé en code review, mais le pattern n'a pas été suffisamment internalisé.

**Root cause :** la règle existe dans project-context.md mais n'est pas matérialisée comme checklist obligatoire dans les Dev Notes des stories. L'agent Dev doit la chercher activement — elle devrait lui sauter aux yeux.

### 2. N+1 queries en Story 4.4

Le composant frontend chargeait les prospects de **tous** les positionings au render de la liste, au lieu de charger à la demande (quand un row est expanded). Non détectable en dev avec 2-3 items — destructeur en prod avec 50 positionings.

**Attrapé en code review adversariale.** Mais aurait dû être anticipé en implémentation.

### 3. `isEditing` persistence entre rows (Story 4.4)

L'état d'édition d'un positioning n'était pas réinitialisé quand l'utilisateur cliquait sur un autre row. L'utilisateur pouvait se retrouver en mode édition sur le mauvais item. Bug UX invisible aux tests automatiques.

### 4. `resetPositionings` accumulation en tests E2E

Les positionings archivés s'accumulaient en DB entre les runs de tests (soft-delete uniquement, pas de hard-delete). Résolu pragmatiquement avec `.first()` sur les locators et en fetchant `?include_archived=true` dans le reset. Pas de hard-delete API disponible — acceptable pour MVP.

## Insights clés

1. **La cohérence UI/code est une contrainte d'architecture, pas une option** — l'Epic 4 le confirme. À documenter formellement pour Epic 5+.

2. **Le pattern SoftDeletes + sous-ressources doit devenir une checklist** — pas juste une note dans project-context.md. Chaque story qui introduit un endpoint de sous-ressource doit vérifier le `.withTrashed()` sur le lookup parent.

3. **La code review adversariale reste le principal filet pour les bugs UX** — N+1 et `isEditing` sont passés à travers les tests fonctionnels et E2E. Elle est non-négociable.

4. **A6 était prématuré, pas oublié** — afficher le nom du positioning dans ProspectDetail n'a de sens complet que quand la chaîne prospect↔interaction↔positioning est entière. Epic 5 complète cette chaîne.

## Décisions produit prises en rétro

### FR21 retiré du scope Epic 5

**Décision :** Le type/subtype d'interaction (FR21) est **retiré du scope Epic 5**.

**Raisonnement :** Le positioning définit déjà la nature de l'interaction. Si le positioning est "Message LinkedIn v3", le type est implicitement "LinkedIn". Ajouter un champ type/subtype en plus violerait le principe "minimal friction" (FR27 : < 1 minute pour logger).

**Champs retenus pour `interactions` :** `prospect_id`, `positioning_id` (nullable), `status` (FR22 : positif/en attente/négatif), `notes` (FR20 : texte libre), `occurred_at`.

### Timeline unifiée par prospect

**Décision :** La timeline d'un prospect (FR25) mélange les **stage transitions** et les **interactions** dans une vue chronologique unique.

- Les transitions de stage = jalons du parcours (lecture seule)
- Les interactions = actions actives (cliquables → détail)
- Distinction visuelle (icônes/couleurs différentes)
