## Ce qui a bien marche

### 1. Refacto `packages/shared` — source de verite unique pour les types

Le refacto post-epic cree `@battlecrm/shared` comme package TypeScript compile partage entre backend et frontend. Les serializers backend (`serializeFunnelStage`, `serializeProspect`, `serializeTransition`) garantissent a la compilation que les reponses API respectent les types definis dans shared. C'est un investissement architectural majeur pour tous les epics futurs.

**Impact :** Tout contrat API deviant sera desormais detecte a `pnpm type-check`, pas en runtime ou en review.

**Patterns etablis :**
- `packages/shared/src/types/*.ts` -> compile vers `.d.ts` via `tsc --build`
- `@battlecrm/shared` dans les `dependencies` backend et frontend
- `#serializers/*` alias dans `apps/backend/package.json` imports
- Serializers : fonctions `serialize*()` avec retour type `SharedType` pour validation TS

### 2. Extraction ProspectDetail + Kanban (Stories 3.8-3.9) — decision architecturale juste

L'ajout de Stories 3.8 (extraction `ProspectDetail`) et 3.9 (Kanban board) n'etait pas prevu initialement. La decision d'extraire le composant avant d'implémenter le Kanban a evite toute duplication : le Drawer du Kanban reutilise `<ProspectDetail>` directement. Architecture clean, zero code duplique.

### 3. Croissance des tests sans flakiness

~105 tests fonctionnels backend + 4 suites E2E Playwright, tous stables. La fondation Japa + Playwright tient bien sur un epic dense (9 stories, 11 jours).

### 4. Patterns etablis et reutilises systematiquement

- `forUser()` scope sur chaque modele -> securite consistante
- Two-step reorder -> reutilise dans `FunnelStagesController.destroy()` pour renumber les positions
- Mutation pattern `useMutation` + `queryClient.invalidateQueries` -> standardise
- `ApiError` -> inline errors, jamais `toast.error()` pour les erreurs de mutation
- `useUpdateProspect()` reutilise pour les changements de stage (pas de hook supplementaire)

### 5. dnd-kit — integration propre pour le Kanban

`@dnd-kit/core` pour le drag-and-drop (choix Epic 1) : drag handle uniquement (a11y), optimistic update + revert sur erreur, `DragOverlay` pour le feedback visuel. Pattern complet documente dans Story 3.9.

## Ce qui a pose probleme

### 1. `prospect_count` -> `prospectCount` — inconsistance detectee tardivement

Le champ `prospectCount` dans les reponses FunnelStage etait nomme `prospect_count` (snake_case) dans les tests fonctionnels mais `prospectCount` (camelCase) dans l'API reelle (Lucid v3 serialise en camelCase par defaut). L'inconsistance n'a ete detectee qu'au refacto `packages/shared`, soit apres la fin des 9 stories — 2 tests ont casse et ont du etre corriges.

**Root cause :** Les tests etaient ecrits sans types partages pour valider la forme des reponses. Le shared package resout le probleme pour l'avenir (les serializers TypeScript garantissent la coherence), mais c'est un bug qui aurait pu etre attrape plus tot si le shared package avait existe des Story 3.2.

### 2. Scope expansion non anticipee — 7 -> 9 stories

Stories 3.8 et 3.9 ont ete ajoutees a `epics.md` le 2026-03-02 alors que l'epic etait en cours. C'est une bonne decision (Kanban est une feature de valeur reelle), mais l'expansion mid-epic revele que le scope initial sous-estimait la surface UI du prospect management.

**Lecon :** En phase de planning Epic 4, anticiper les composants partages et les vues alternatives des l'ecriture des stories.

### 3. `stageTransitions()` + stages soft-deleted — comportement subtil

En Story 3.6, `preload('fromStage')` peut retourner `null` si le stage a ete soft-deleted apres la transition. Le workaround (`as any` cast pour `.withTrashed()` dans un preload) est fonctionnel mais pas elegant. Acceptable pour MVP.

### 4. UUID validation — protection contre les 500

Les params UUID non-valides envoyes a PostgreSQL causent des erreurs `invalid input syntax for type uuid` -> 500. Decouverte documentee en memoire mais necessite une vigilance systematique a chaque nouveau endpoint.

### 5. SoftDeletes + sous-ressources — ordre des scopes

`stageTransitions()` qui cherche un prospect parent doit inclure `.withTrashed()` pour que les prospects archives restent accessibles. Sans ca, un prospect archive donne un 404 silencieux sur sa sous-ressource.

## Insights cles

1. **Le refacto shared est un game-changer** — la verification des contrats API a la compilation change fondamentalement le rapport aux bugs d'API. Decision architecturale la plus impactante de cet epic.

2. **Lucid v3 camelCase par defaut est une source de confusion persistante** — l'architecture doc dit snake_case mais l'API retourne camelCase. Le shared package centralise la verite, mais la divergence de doc merite une note dans `architecture.md`.

3. **La code review adversariale reste non-negociable** — les bugs UX/frontend (inline errors vs toast, a11y dnd-kit, optimistic revert) ne sont pas detectables par les tests automatiques.

4. **Les stories de refacto/extraction ont autant de valeur que les features** — sans Story 3.8, Story 3.9 aurait ete deux fois plus complexe. Ne pas negliger les stories "technique" dans les plannings.

5. **Serializer + shared types doivent etre en place des la premiere story d'un epic** — ne pas attendre un refacto post-epic pour centraliser les types.

Le package shared doit etre builde avant le type-check des apps. Actuellement pas automatise dans le root `package.json`. A corriger avant Epic 4 (Action Item A1).

## Decisions architecturales confirmees pour la suite

| Decision | Detail |
|---------|--------|
| Shared types | `@battlecrm/shared` — workspace package, compile vers `.d.ts`, `exports.types` |
| Serializers | `#serializers/*` — fonctions TypeScript-typed par entite, dans backend uniquement |
| API camelCase | Lucid v3 serialise camelCase — tous les types shared refletent ca (`funnelStageId`, `deletedAt`, etc.) |
| `$extras` aggregates | `withCount('relation', cb)` -> `$extras.relation_count` -> mapper manuellement dans serializer |
| UUID validation | Toujours valider le format UUID avant `.where('id', param)` sur colonne uuid |
| Soft delete + sous-ressources | `.withTrashed()` obligatoire sur les lookups de parents archives |
| Optimistic updates Kanban | `optimisticOverrides: Record<string, string>` + cleanup useEffect quand query se met a jour |
| Drag-and-drop | `@dnd-kit/core` — `useDraggable` sur handle uniquement, `useDroppable` sur colonnes, `DragOverlay` |
